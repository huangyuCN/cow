# COW 需求说明重排设计

## 1. 背景

当前仓库中的 [MVP_REQUIREMENTS.md](/Users/huangyu/work/golang/src/cow/MVP_REQUIREMENTS.md) 已经包含较完整的方向判断，但内容同时混合了需求、设计草图与部分实现导向表述，存在以下问题：

- 评审难以快速判断第一版范围是否合理；
- 后续实现者难以区分“已经确认的需求边界”和“尚未确认的实现细节”；
- `MVP` 范围、误用约束、验收标准不够集中，不利于继续拆 spec 与测试。

本设计文档的目标不是设计运行时实现，而是定义“如何把现有需求说明重排成更适合决策与后续落地的版本”。

## 2. 本次设计目标

本次改写目标如下：

1. 保持 `MVP_REQUIREMENTS.md` 作为需求说明，而不是实现设计稿。
2. 同时服务决策者与后续实现者。
3. 将 `MVP` 收敛为“纯库级、正确性优先”的最小可验证范围。
4. 将对外使用模型、安全边界、误用约束、验收标准写成明确规则。
5. 为后续单独编写实现 spec 与计划文档预留清晰接口。

## 3. 已确认需求边界

经本轮头脑风暴确认，以下边界已固定：

- 文档类型：需求说明，而不是实现设计稿。
- 目标读者：同时面向作者本人和协作者/评审。
- `MVP` 主验收场景：纯库级场景。
- 第一优先目标：正确性。
- 安全取向：严格安全优先。
- 写入模型：`MVP` 默认采用组件级受控写入。
- `Savepoint / RollbackTo`：属于 `MVP` 必做能力。
- `Savepoint` 语义：支持多层，但仅保证栈式使用正确。
- 误用处理：默认明确错误，`Debug` 模式增加更激进检测。
- `DirtySet`：可有简版，但不是第一优先验收项。
- 代码生成器：保留为后续演进方向，不属于 `MVP` 交付重点。
- 标准入口：`cow.Run(ctx, store, fn)` 作为 `MVP` 唯一主路径。

## 4. 文档结构设计

需求说明应重排为以下结构：

1. 背景与问题定义
2. 适用边界与运行前提
3. 核心需求与已确认决策
4. `MVP` 范围与非 `MVP` 范围
5. 对外使用模型与行为语义
6. 安全边界与误用约束
7. 验收标准
8. 后续演进路线

该结构的目的如下：

- 前四节帮助评审快速判断边界、价值与第一版大小；
- 中间两节把使用契约和禁止用法写清；
- 验收标准直接为后续测试与 benchmark 提供输入；
- 演进路线保留长期方向，但不污染 `MVP`。

## 5. 关键写法要求

### 5.1 需求层优先

`MVP_REQUIREMENTS.md` 应优先回答“做什么、为什么、做到什么算完成”，而不是“内部怎么实现”。因此应减少以下内容：

- 过细的数据结构示意；
- 具体接口签名的展开；
- 实现层 clone 细节与代码片段；
- 宿主适配的具体做法。

### 5.2 行为契约优先

需求说明必须将以下内容写成规则而非建议：

- 标准事务入口；
- 组件级受控写入；
- `Savepoint` 的栈式语义；
- 禁止绕过事务直接写已提交态；
- 禁止把可变引用带出有效作用域继续使用。

### 5.3 验收标准可测试

验收标准必须能自然映射到后续测试，而不能停留在抽象口号。例如：

- 成功提交后整体可见；
- 返回错误或发生 `panic` 后整体不可见；
- `Savepoint` 回滚仅撤销检查点之后的修改；
- 根事务在局部回滚后仍可继续并最终提交；
- 与全量 `deepcopy` 方案存在可复现实验对比。

## 6. 文档改写结论

本次改写不改变核心方向判断，只改变文档表达方式：

- 保留“单聚合、串行托管、宿主无关”的基本定位；
- 保留运行时 plain Go struct、持久化 DTO 分离的边界；
- 保留 `Savepoint` 属于需求承诺；
- 将文档从“需求 + 设计混排”重构为“需求说明主文档”；
- 将实现细节、API 收敛、错误模型等推迟到后续 spec。

在本设计文档得到审阅后，后续应按以下顺序推进：

1. 审阅并确认新的 [MVP_REQUIREMENTS.md](/Users/huangyu/work/golang/src/cow/MVP_REQUIREMENTS.md)。
2. 进入实现设计文档编写，单独收敛核心 API、`Store` 契约、错误模型、`Savepoint` 规则与测试矩阵。
3. 基于设计文档再拆实现计划，进入 `TDD`（Test-Driven Development，测试驱动开发）流程。
