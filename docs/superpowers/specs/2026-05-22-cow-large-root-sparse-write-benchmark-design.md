# COW 大根稀疏写 Benchmark 设计

## 1. 背景

当前仓库中的 `COW`（Copy-On-Write，写时复制）运行时已经完成：

- 显式 `TxSession` 主模型切换；
- 组件级 + 容器级 `COW` 主写路径；
- `Begin()` 默认只读、首次写升级的框架层固定分配优化。

最近一轮 benchmark 已经证明：

- 空事务 / 只读事务方向上的框架固定成本确实下降了；
- 但当前 benchmark 仍主要围绕一个非常小的示例根结构；
- 现有结果还不足以回答“当根规模真正变大、每次只改一个局部组件时，这条 `COW` 路线是否会逐渐优于整根 `DeepCopy`”。

当前 benchmark 的主要局限有三点：

1. 根结构过小，只有极少数组件；
2. 样例主要用于语义演示，而不是规模验证；
3. 还无法观察“根规模增大后，`COW` 与 `DeepCopy` 的相对差距是否收敛”。

因此，下一轮 benchmark 设计不再优先扩实现能力，而是要先补一组**大根稀疏写**对照基准，用来判断当前框架方向本身是否成立。

## 2. 本次设计目标

本轮只覆盖以下目标：

1. 构造一组专门用于 benchmark 的大根测试模型。
2. 在相同业务动作下，对比 `COW` 与整根 `DeepCopy`。
3. 让根规模按 `16 / 64 / 256` 三档递增。
4. 固定为“每次事务只改 1 个组件”。
5. 用结果判断：随着根规模增大，`COW` 相对 `DeepCopy` 的差距是否收敛，甚至反超。

## 3. 本次设计不覆盖的范围

本轮不覆盖以下内容：

- 只读事务 benchmark 扩展；
- 同事务热点重复写 benchmark；
- 改 2 个或更多组件的稀疏写矩阵；
- 不同容器类型（例如 `slice`）对照；
- 不同组件位置分布（随机位置、中间位置）；
- `Savepoint` benchmark。

也就是说：

- 本轮只回答一个核心问题：
  - 当根规模变大且每次只改 1 个组件时，当前 `COW` 路线是否开始显现结构性优势。

## 4. 方案比较

### 方案 A：最小验证面的大根稀疏写 benchmark

- 组件规模固定为 `16 / 64 / 256`
- 每次只改第 `0` 个组件
- 每个组件只有：
  - 一个标量字段
  - 一个固定大小的 `map`
- 写入模式固定为：
  - `1` 次标量写
  - `1` 次 `map` 写

优点：

- 变量最少；
- 最容易解释结果；
- 最适合第一轮验证设计方向。

缺点：

- 信息面较窄；
- 还不能说明只读事务和热点写场景。

### 方案 B：一步到位的稀疏写矩阵

- 在方案 A 基础上，再扩出：
  - 改 `1` 个 / `2` 个组件；
  - 不同组件位置；
  - 不同 map 体量。

优点：

- 一轮信息更多。

缺点：

- 第一轮组合数膨胀太快；
- 一旦结果异常，难以归因；
- 不利于快速判断路线是否成立。

### 方案 C：直接构造复杂业务近似模型

- 根组件不完全同构；
- 写入动作更贴近真实业务。

优点：

- 更接近最终使用场景。

缺点：

- 结果解释复杂；
- 过早引入过多业务变量；
- 不适合作为第一轮方向验证 benchmark。

### 推荐方案

推荐采用**方案 A**。

原因：

- 当前最重要的是验证“方向是否成立”，不是一次性模拟全部真实负载；
- 只有先在最小变量集上看清趋势，后续补只读、热点写和更真实模型才有意义。

## 5. Benchmark 数据模型

### 5.1 专用 benchmark 根类型

建议新增一套专门用于 benchmark 的测试模型，不直接复用现有 `testRoot / Bag / Quest` 小样例。

推荐结构如下：

```go
type benchSparseRoot struct {
    Comps []*benchSparseComp
}

type benchSparseComp struct {
    Gold  int
    Items map[int]int
}
```

语义如下：

- `benchSparseRoot` 表示可按规模扩展的大根；
- `Comps` 的长度决定根规模；
- `benchSparseComp` 表示同构组件；
- `Gold` 用于标量写；
- `Items` 用于容器写。

### 5.2 规模梯度

第一轮固定三档组件规模：

- `16`
- `64`
- `256`

选择这三档的原因是：

- `16` 作为相对较小但已脱离当前玩具样例的规模；
- `64` 作为中档观察点；
- `256` 用于放大根规模差异，尽量观察是否出现拐点。

### 5.3 单组件体量

第一轮中，每个组件内部只保留一个固定大小的 `map[int]int`。

固定大小为：

- `128` 个键值对。

原因：

- 比 `16 / 32 / 64` 更不容易被框架固定成本淹没；
- 又没有大到把 benchmark 变成纯容器复制测试；
- 适合作为“中等体量”的第一轮基线。

### 5.4 组件位置

第一轮固定为：

- 每次只修改 `Comps[0]`。

这样做不是因为第 `0` 个组件更真实，而是为了：

- 避免第一轮引入随机位置噪音；
- 让结果更稳定；
- 让问题归因更集中在复制策略本身。

## 6. Benchmark 命名方案

第一轮采用独立 benchmark 名称，不使用 `b.Run(...)` 子基准。

建议新增以下基准：

- `BenchmarkCowSparseWrite16`
- `BenchmarkCowSparseWrite64`
- `BenchmarkCowSparseWrite256`
- `BenchmarkDeepCopySparseWrite16`
- `BenchmarkDeepCopySparseWrite64`
- `BenchmarkDeepCopySparseWrite256`

这样做的好处是：

- 基准结果最直观；
- `benchstat` 对比更直接；
- 后续若扩展“改 2 个组件”或“不同 map 体量”，命名也容易平滑扩展。

## 7. Benchmark 写路径设计

### 7.1 COW 组

每轮 benchmark 固定执行以下流程：

1. 构造带 `N` 个组件的大根；
2. `Begin(...)`；
3. 获取第 `0` 个组件的可写视图；
4. 对该组件执行两次写：
   - `Gold += 1`
   - `Items[0] += 1`
5. `Commit()`。

### 7.2 DeepCopy 对照组

每轮 benchmark 固定执行以下流程：

1. 构造带 `N` 个组件的大根；
2. 对整根做一次深拷贝；
3. 获取第 `0` 个组件；
4. 执行与 `COW` 组完全相同的两次写：
   - `Gold += 1`
   - `Items[0] += 1`

### 7.3 对齐约束

两组 benchmark 必须严格对齐以下条件：

- 使用同一份大根数据形状；
- 都只改第 `0` 个组件；
- 都执行 `1` 次标量写 + `1` 次 `map` 写；
- 都不引入额外读路径；
- 都不加入第二次重复写；
- 都不混入 `Savepoint`。

这样 benchmark 的差异才能主要解释为：

- `DeepCopy` 多承担整根复制；
- `COW` 多承担事务框架、首次写升级、组件级复制、容器级复制。

## 8. 构造辅助函数

为了保持 benchmark 代码清晰，建议至少补两类辅助函数：

### 8.1 大根构造函数

负责：

- 按给定组件数量构造 `benchSparseRoot`；
- 为每个组件生成固定 `128` 项的 `map`；
- 保证每个组件都拥有独立容器实例。

### 8.2 整根深拷贝函数

负责：

- 为 `DeepCopy` 对照组提供整根克隆能力；
- 深拷贝所有组件；
- 深拷贝每个组件的 `map`。

必要时还可以补：

- `mutableSparseComp(...)`
- `mutableSparseItems(...)`

但第一轮不要求先把 benchmark 专用写路径抽象得过度泛化，重点是结果干净、结构清楚。

## 9. 结果判读标准

### 9.1 第一层：趋势是否支持方向

重点观察：

- 当规模从 `16 -> 64 -> 256` 增大时；
- `DeepCopy` 的 `ns/op`、`B/op`、`allocs/op` 是否明显上升；
- `COW` 也会上升，但理想情况下升幅更慢。

若出现这种趋势，说明方向成立：

- 根越大、写越稀疏，`COW` 的相对价值越高。

### 9.2 第二层：是否出现拐点

理想预期是：

- `16` 组件时，`COW` 可能仍然落后；
- `64` 组件时，差距开始缩小；
- `256` 组件时，差距进一步收敛，甚至反超。

本轮**不强制要求**第一轮就出现绝对反超。

只要能够观察到：

- 差距随着规模扩大而收敛，

就可以把结论判为：

- 当前路线方向正确，但实现仍有优化空间。

### 9.3 第三层：失败信号

若出现以下现象，则要认真怀疑当前实现形态：

- `DeepCopy` 随规模明显变重；
- 但 `COW` 也以近似同样速度变重；
- `COW` 的 `B/op` / `allocs/op` 长期没有相对优势；
- 到 `256` 组件仍看不到差距收敛。

这通常意味着以下某类成本可能过高：

- 根级首次写升级；
- `dirty / cloned` 的 map 管理；
- 写入口判断成本；
- 当前 benchmark 数据模型仍不足以体现真实目标场景。

### 9.4 结果结论分级

本轮 benchmark 的验收结论建议分成三类：

1. `方向正确`
   - 差距随规模扩大而收敛，或出现反超。
2. `方向可能正确，但实现还有明显固定成本`
   - 有收敛，但收敛幅度有限。
3. `方向存疑`
   - 基本看不到收敛，甚至相对更差。

## 10. 测试与归档要求

本轮实现后，至少需要运行：

```bash
go test ./... -run '^$' -bench 'Benchmark(CowSparseWrite|DeepCopySparseWrite)' -benchmem -count=1
```

以及与上一轮已归档或已保存基线做对比。

结果反馈时至少应包含：

- 各规模下 `COW` 与 `DeepCopy` 的 `ns/op`
- `B/op`
- `allocs/op`
- 相对变化趋势
- 对趋势的结论分级

若用户确认保留结果：

- 将表格与元数据追加到 `docs/superpowers/benchmarks/cow-mvp-benchmark.md`。

## 11. 结论

下一轮不应继续凭当前小样例讨论 `COW` 与 `DeepCopy` 的最终胜负，而应先补一组专门的大根稀疏写 benchmark。

这组 benchmark 的核心特征是：

- 大根；
- 同构组件；
- 中等大小 `map`；
- 每次只改 1 个固定组件；
- `COW` 与 `DeepCopy` 业务动作完全对齐。

只有先拿到这组结果，才能更有把握地回答：

- 当前框架设计方向是否正确；
- 还是只是局部优化看起来成立，但总体路线并不会在真实规模下获益。
